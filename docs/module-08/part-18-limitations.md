## Part 18: Limitations

**Hyperparameter tuning on a single fold.** The Optuna studies tune on fold 3 only (train 2022-2024, validate 2025). Parameters that are optimal for predicting 2025 experience may not be optimal for predicting 2026. The correct production practice is to re-tune quarterly using the most recent fold. Parameters from a year-old tuning run are likely still good — CatBoost depth and learning rate are not sensitive to the specific accident year — but re-tuning quarterly is cheap and eliminates the risk.

**Conformal calibration assumes year-over-year exchangeability.** The conformal predictor calibrates on 2024 claims and is tested on 2025 claims. If claims inflation was 15% in 2025, the 2024 calibration residuals are systematically smaller than the 2025 residuals, and the 90% interval threshold set from 2024 data will undercover in 2025. This is not a theoretical edge case: UK motor severity inflation ran at 20-32% in 2022-23. For a book experiencing material inflation, use a combined 2023-2024 calibration set to smooth the year-over-year shift, or recalibrate the conformal predictor monthly.

**The rate optimiser works on a sample.** Stage 9 uses `min(5000, len(df_test))` policies for the optimisation. The SLSQP solver is analytical and runs in under a second even for 50,000 policies — there is no computational reason to sample. Use the full renewal portfolio in production.

**Pure premium as input to the optimiser.** The optimiser receives `pure_premium` as both `technical_price` and `expected_loss_cost`. This implicitly assumes the model is perfectly calibrated (Stage 8.5 checks this). After a balance correction, `expected_loss_cost` should be the corrected predictions, not the raw model output. The pipeline passes `freq_pred_final` (corrected if needed) to the pure premium computation — verify this connection is intact if you modify Stage 8.5.

**The efficient frontier uses sequential point solves.** The frontier traces 12 points across LR targets from 0.68 to 0.78. Each solve is independent. Near the boundaries of the feasible region (very tight LR targets or volume floors), the solver may not converge for some points. Check `frontier_df` for NaN values in the converged column before presenting it to the pricing committee. Drop non-converged points rather than interpolating — a non-converged result is not a valid frontier point.

**The audit table uses `run_date` as the primary key.** Multiple runs on the same date (e.g., a morning run and an afternoon run after fixing a data issue) produce rows with identical `run_date` values. Distinguish them using `freq_model_run_id`, which is unique per run. If you need a true primary key, add `pipeline_run_uuid = str(uuid.uuid4())` to the configuration cell and include it in the audit record.

**SHAP relativities are computed on the test set.** The `SHAPRelativities` object computes values from the test year's feature distribution (2025). If the 2025 book mix is skewed — for example, more young drivers in London than in 2022-2024 — the regional relativities will reflect that skew rather than the prospective portfolio composition. For rate-setting purposes, compute SHAP relativities on the prospective renewal portfolio, not the historical test set. Pass the renewal portfolio features to `SHAPRelativities` in Stage 7 when it becomes available.

**GLM components are not included.** Some UK motor pricing teams require a GLM sign-off alongside the GBM — a parallel GLM run that validates the GBM's factor relativities are in the expected direction and magnitude. This pipeline does not include a parallel GLM. Adding one is straightforward: fit a Poisson GLM using `statsmodels.formula.api.glm()` in Stage 6 using the same features table, log it to MLflow, and compare its factor relativities to the SHAP relativities in Stage 7. The comparison is the validation output the pricing committee requires.
